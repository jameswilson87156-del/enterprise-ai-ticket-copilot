# Codex 项目工作规则

## 1. 项目定位

这是一个面向零基础用户的 Claude + Codex 本地协作模板。Codex 负责根据已确认的 `docs/` 和 `TODO.md` 完成小范围实现、运行验证、更新记录。Claude 与 Codex 不会自动互聊，跨 Agent 信息必须写入项目文件。

开始工作前必须读取：

- `AGENTS.md`
- `docs/PRD.md`
- `docs/DESIGN.md`
- `docs/ARCHITECTURE.md`
- `TODO.md`
- `HANDOFF.md`

## 2. 开发边界

- 每轮只能完成一个名称明确、范围可验收的 TODO。
- 修改前先检查项目目录、相关源码、现有文档；存在 Git 仓库时查看 `git status` 和当前 diff。
- 不得顺手实现下一个 TODO，不得把小任务扩展为全项目重构。
- 不得实现 `docs/PRD.md`“暂不做事项”中的内容。
- 新增生产依赖、数据库、付费服务、外部账号或破坏性变更前，必须获得用户明确同意。
- 用户已有的无关改动必须保留；不覆盖、不回滚、不混入本轮修改。

## 3. 每次任务流程

1. 审查现状：读取规则、文档、任务、交接记录和相关代码。
2. 确认范围：复述目标、涉及文件、不可破坏项和验收方式。
3. 制定最小计划：只覆盖当前一个任务。
4. 实施修改：保持改动集中，不重写无关代码。
5. 执行验证：运行实际存在且适用的测试、构建、lint、类型检查或启动检查。
6. 检查结果：查看错误输出和 diff，不只根据命令退出码猜测。
7. 更新文档：同步 `TODO.md`、`HANDOFF.md`，必要时更新 `docs/` 或 `README.md`。
8. 汇报：列出修改文件、验证命令、实际结果、未验证内容和剩余风险。

## 4. 测试要求

- 修改后必须运行项目中可用且与本轮相关的验证命令。
- 如果没有自动化测试，必须提供可重复的手工验收步骤，并明确标记“未运行自动化测试”。
- 命令未执行、失败或因环境受阻时，必须如实说明。
- 不能编造测试输出，不能把“代码看起来没问题”写成“测试通过”。
- 验证失败时，不得把 TODO 标记为完成。

## 5. Git 提交规则

- 修改前检查 `git status`；提交前检查 `git diff`。
- 一个提交只包含一个已经验证的小任务。
- 提交信息应说明本次行为，例如 `feat: 添加登录表单校验`。
- 未经用户明确授权，不执行 `git push`、不创建远程仓库、不部署生产环境。
- 不使用 `git reset --hard`、强制覆盖或其他破坏性命令处理用户改动。
- 不把以下内容加入 Git：
  - `.env`、`.env.*` 的真实配置；
  - 密码、Token、API Key、证书和个人隐私；
  - `node_modules/`、`dist/`、`target/`、`build/`；
  - `.cache/`、临时目录、编辑器缓存、日志和系统生成文件。

## 6. 文档更新要求

- 需求变化：更新 `docs/PRD.md`。
- 页面、交互或状态变化：更新 `docs/DESIGN.md`。
- 技术栈、目录、接口、数据或运行方式变化：更新 `docs/ARCHITECTURE.md`。
- 任务状态变化：更新 `TODO.md`。
- 每轮实现、测试、阻塞与审查结果：追加到 `HANDOFF.md`，不覆盖历史记录。
- 文档只能描述真实现状；未确认内容标记为“待确认”。

## 7. 禁止事项

- 禁止一次性完成整个大项目。
- 禁止删除重要文件或清理用户数据，除非用户明确指定并确认影响。
- 禁止自动安装软件、登录账号、修改系统环境变量或替用户购买服务。
- 禁止把 Claude/Codex 描述成会自动互聊。
- 禁止伪造测试、构建、部署或 Git 提交成功。
- 禁止在未理解影响时执行管理员命令、递归删除或全局配置修改。
