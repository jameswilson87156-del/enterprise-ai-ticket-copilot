# 面试讲解指南

## 项目一句话

Enterprise AI Ticket Copilot 是一个 Java + Vue 的企业内部工单辅助处理工作台。它用规则引擎辅助分类、基于分类和关键词评分的知识库匹配、模板化处理建议草稿，串起工单流转和知识沉淀；所有建议都需要人工确认后才能进入处理流程或知识库。

这里的 Copilot 是产品名，表示“面向客服 / 运维人员的辅助处理工作台”，不代表项目已经接入真实大模型。

## 可以突出什么

- 后端：Spring Boot 3、Java 17、MyBatis-Plus、MySQL 五表闭环。
- 工程质量：Bean Validation、全局异常处理、Swagger/OpenAPI、知识发布幂等保护、17 项 Controller/Service 自动化测试。
- 前端：Vue 3 + Vite 双列 Copilot 工作台，不是普通 CRUD 表格后台。
- 规则辅助设计：当前不接真实 LLM，通过规则分类、关键词匹配和模板生成保证可解释、低延迟、无外部 API 依赖。
- 安全边界：不自动授权、不自动回滚、不调用外部系统，所有建议草稿保留“待人工确认”。
- 产品闭环：提交工单 -> 规则辅助分类 -> 知识命中 -> 建议草稿 -> 人工处理 -> 状态历史 -> 知识草稿 -> 人工确认发布。

## 技术问答准备

### 面试官问：你这个 AI 是怎么实现的？

标准回答：

本项目没有接入真实 LLM，也没有训练模型。分类采用规则引擎和关键词权重，知识匹配采用分类和关键词评分，建议内容采用模板化草稿。这样做是为了在演示项目中保证可解释、低延迟、无外部 API 依赖，并且所有建议都需要人工确认后才进入处理流程或知识库。

可以补充：

- 规则分类逻辑在 `RuleClassificationService`。
- 知识匹配逻辑在 `KnowledgeMatchingService`。
- 建议草稿逻辑在 `RecommendationTemplateService`。
- 审计记录保存在 `generation_record`，状态流转保存在 `ticket_status_history`。

### 面试官问：为什么不用大模型？

标准回答：

这个项目的目标是先验证企业工单闭环和人机协同边界。企业内部工具通常更重视可解释、可审计、可复现和数据不外发。规则引擎和模板可以先把流程跑通，避免外部 API 成本、网络波动和敏感数据风险。如果后续接企业私有模型或受控检索增强，也会保留人工确认和审计链路。

### 面试官问：知识匹配是不是向量检索？

标准回答：

不是。当前没有 embedding，也没有向量数据库。知识匹配是基于工单分类、标题、描述、错误日志和知识条目关键词的评分公式。这样实现简单、可解释，面试时可以直接说明命中原因。后续如果要扩展向量检索，会作为新的 V2 能力重新设计，不会把当前版本说成已经实现。

### 面试官问：这个系统能不能生产使用？

标准回答：

当前仓库是个人作品集 / 实习简历项目，不是生产系统。它已经具备后端分层、MySQL 数据闭环、状态流转、审计记录、统一异常响应、Swagger 文档和自动化测试证据，但还缺少生产级鉴权、权限模型、CI/CD、监控告警、数据脱敏和部署方案。所以我会把它定位为“可演示的企业工单工作流系统原型”，不夸大成生产上线系统。

### MySQL 表怎么设计？

- `support_ticket`：工单主表。
- `knowledge_article`：知识库文章和草稿。
- `ticket_ai_analysis`：规则分类、置信度、知识命中和建议草稿。
- `generation_record`：规则/模板生成记录，方便审计。
- `ticket_status_history`：状态流转历史。

### 前端为什么做成 Copilot 工作台？

目标用户是内部支持人员和运维，不需要营销落地页，也不适合聊天页。双列主工作区将工单队列与入口、工单上下文与建议草稿分组展示，并把知识沉淀和状态历史放在后续区域，适合高频处理和演示。

## 可展示的接口

- `POST /api/tickets`
- `GET /api/tickets`
- `GET /api/tickets/{id}`
- `GET /api/tickets/{id}/ai-analysis`
- `POST /api/tickets/{id}/status`
- `POST /api/tickets/{id}/knowledge-draft`
- `POST /api/tickets/knowledge/{articleNo}/confirm`
- Swagger UI：`http://localhost:8080/swagger-ui/index.html`

## V2 方向

- 企业内网私有模型或受控检索增强评估。
- 更精细的关键词权重、分类权重和人工采纳率统计。
- 多角色审核流。
- 操作审计与权限控制。
- SLA、告警联动和报表分析。
- 但仍保留人工确认，不做自动生产变更。
